# 完整项目设计方案

## 1. 项目定位

项目名称：基于 3D 深度学习的肺结节智能筛查与医患双端协同系统

设计目标：
1. 提升医生阅片效率，降低漏诊风险。
2. 构建“AI 预测 -> 医生审核 -> 患者理解 -> 随访提醒”的闭环。
3. 采用可替换的开源模型底座，快速形成可部署原型。

## 2. 角色与核心场景

角色：
- 医生（影像科/呼吸科）
- 患者
- 管理员

核心场景：
1. 医生上传 CT 并触发 AI 推理。
2. 系统回传结节坐标、体积、检测置信度与辅助风险等级，按风险排序患者。
3. 医生在 3D 阅片界面确认后发布结构化报告。
4. 患者查看图文报告并接收复查提醒。

## 3. 总体架构

### 3.1 分层架构

- 表现层：React 双端（医生端 PC + 患者端 H5）
- 业务层：FastAPI（鉴权、患者管理、报告流转、AI 任务编排）
- 算法层：AI Engine（预处理、分割、分类集成）
- 数据层：PostgreSQL + MinIO + Redis

### 3.2 部署拓扑

- Nginx 统一入口：
  - `/api/*` -> Backend
  - `/ai/*` -> AI Engine
  - `/` -> 前端静态资源
- Redis 作为任务队列和缓存
- PostgreSQL 存业务结构化数据
- MinIO 存原始 CT 和预处理缓存文件

## 4. 关键业务流

### 4.1 CT 检查处理主链路

1. 医生创建检查单并上传 CT 文件到对象存储。
2. 后端写入 `studies` 记录并创建 `ai_jobs`（PENDING）。
3. Celery Worker 调用 AI Engine 推理。
4. AI Engine 返回：
   - 结节列表（坐标、直径、体积）
   - 每个结节的检测置信度
   - 全局风险等级
5. 后端写入 `nodules`、`ai_predictions`，更新分诊看板排序。
6. 报告服务生成结构化草稿，医生审核签发。
7. 患者端收到通知并展示通俗化报告。

### 4.2 随访提醒链路

1. 医生或系统设置随访周期（3/6/12 个月）。
2. 定时任务扫描到期计划。
3. 生成 `followup_notifications` 并通过站内信/短信推送。
4. 患者确认后记录回执与后续检查计划。

## 5. 模块设计

### 5.1 前端模块（React）

医生端：
- 智能分诊看板（风险倒序、红黄绿标签）
- 3D 阅片室（切片滚动、窗宽窗位、病灶框选）
- 结构化报告编辑器（自动草稿 + 审核签发）

患者端：
- 健康档案时间轴
- 图文报告解读（风险信号灯）
- 随访计划与提醒

### 5.2 后端模块（FastAPI）

- `auth`：JWT 登录鉴权、角色权限
- `patients`：患者信息与就诊关系
- `studies`：检查单、文件记录、状态流转
- `ai_jobs`：推理任务创建、重试、状态查询
- `triage`：风险排序和高危标记
- `reports`：草稿生成、版本管理、发布
- `followups`：复查周期与提醒消息

### 5.3 AI 引擎模块（模型可插拔）

- `preprocess`：SimpleITK 读入 -> 方向统一 -> 重采样 -> 强度归一化
- `detection`：MONAI 3D lung nodule bundle 输出候选结节框与检测分数
- `risk stratification`：结合结节大小、数量、检测分数和规则生成系统级辅助风险评分
- `postprocess`：坐标还原（origin/spacing）、体积计算、阈值分级

## 8. AI 接入策略（模型适配层二次封装）

开源仓库具备训练与预处理能力，推理服务化需补齐三类能力：
1. 模型权重管理：统一权重目录与版本号。
2. 推理入口封装：按 Study 读取 CT，输出统一 JSON。
3. 服务化和幂等：支持 job_id 级别重试与重复调用去重。

建议新增接口：
- `POST /v1/predict`
- `GET /v1/jobs/{job_id}`

输入：对象存储 URI、origin/spacing、患者检查元数据。
输出：结节数组、风险等级、推理耗时、模型版本。

## 9. 性能与可靠性指标

- 单例推理时延目标：`P50 < 8s`，`P95 < 20s`（GPU 推理）
- 系统可用性：`>= 99.5%`（教学/科研环境）
- AI 任务重试：最多 3 次，指数退避
- 数据一致性：报告发布前需校验 AI 结果版本未变化

## 10. 安全与合规

- JWT + RBAC（医生/患者/管理员）
- 文件上传白名单与病毒扫描（可后续接入）
- 关键行为审计：报告签发、删除、权限变更
- 脱敏策略：导出和日志中隐藏姓名、身份证、手机号
- HTTPS + 对象存储私有桶

## 11. 项目实施计划（8 周）

第 1 周：需求冻结与数据库/API 定稿

第 2-3 周：
- 后端基础服务（鉴权、患者、检查、报告）
- 对象存储与上传链路

第 4-5 周：
- AI 引擎服务化改造
- 任务队列与异步推理接入

第 6 周：
- 医生端分诊看板 + 阅片集成
- 患者端报告与随访页面

第 7 周：
- 联调与性能压测
- 故障演练（AI 服务不可用、任务堆积）

第 8 周：
- 文档收敛、演示环境部署、答辩材料

## 12. 主要风险与缓解

1. 模型版本兼容性风险（旧版 Keras/TensorFlow）
- 缓解：AI 服务独立镜像，冻结依赖与 CUDA 版本。

2. 医学影像前端渲染复杂度高
- 缓解：先交付 2D 切片 + BBox，3D 渲染分阶段上线。

3. 推理时延波动
- 缓解：预处理缓存、GPU 资源隔离、任务优先级队列。
