# 完整项目设计方案

## 1. 项目定位

项目名称：DeepLung 肺结节智能辅助检出与医患协同随访系统。

本项目面向《医学人工智能》课程大作业，设计一个基于肺部 CT 的 AI 医疗软件原型。系统以肺结节辅助检出为核心，围绕医生端分诊、报告签发、患者端报告查看、消息沟通与随访解释，构建“AI 检测 - 医生审核 - 患者理解 - 随访管理”的完整业务闭环。

系统定位为课程原型和教学展示平台，不作为临床诊断系统直接使用。所有 AI 输出仅作为辅助信息，最终判断以医生签发报告为准。

## 2. 设计目标

1. 提升医生在肺部 CT 场景下的初筛和风险排序效率。
2. 构建“检查登记 - AI 分析 - 报告签发 - 患者查看 - 随访提醒”的完整流程。
3. 支持真实模型与 mock 模式双通路，保证课堂演示稳定。
4. 体现医学 AI 产品的完整设计思路，而不仅是单点检测算法。

## 3. 角色与核心场景

系统包含三类角色：

- 医生：上传 CT、查看 AI 结果、签发报告、发送患者消息。
- 患者：查看已发布报告、接收随访提醒、通过智能助手获取解释。
- 管理员：用于账号初始化与演示环境维护。

核心场景如下：

1. 医生为患者创建 CT 检查记录并触发 AI 推理。
2. 系统返回结节坐标、直径、检测分数和辅助风险等级，并更新风险排序列表。
3. 医生在工作台审核结果，编辑 impression 与 recommendation，签发正式报告。
4. 患者在患者端查看报告、接收医生消息，并获得通俗化解释。

## 4. 总体架构

### 4.1 分层架构

- 表现层：React 前端，包含医生端和患者端界面
- 业务层：FastAPI 后端，负责认证、检查记录、AI 任务编排、报告与消息接口
- 算法层：AI Engine，封装 mock 检测器和 MONAI bundle 检测器
- 数据层：SQLite 持久化业务数据，本地文件路径用于 CT 读取

### 4.2 架构特点

1. 前后端分离，页面与业务逻辑解耦。
2. AI 服务独立部署，业务服务通过 HTTP 调用模型服务。
3. 检测器采用适配器模式，统一输出字段，便于模型替换。
4. 患者问答模块可切换 mock 或外部大模型接口。

## 5. 关键业务流程

### 5.1 CT 检查主链路

1. 医生登录后输入患者 ID 和 CT 文件路径。
2. 后端创建 `studies` 记录，并生成 `study_id`。
3. 医生触发 AI 推理，后端创建 `jobs` 记录。
4. 后端后台任务调用 AI Engine 执行推理。
5. AI Engine 返回结节列表、风险分值、风险等级和文本摘要。
6. 后端写入任务结果，并同步更新 `patient_triage` 风险排序表。
7. 医生审核 AI 结果后签发报告，报告写入 `reports` 表。
8. 患者端可查看正式报告和随访日期。

### 5.2 患者解释与随访链路

1. 患者登录后查看自己的报告列表和当前报告详情。
2. 系统展示风险灯、摘要、建议、结节信息和随访日期。
3. 患者可查看医生发送的消息记录。
4. 患者围绕当前报告发起智能问答，系统结合报告内容返回解释性回答。

## 6. 模块设计

### 6.1 前端模块

医生端包括：

- 登录页
- 风险分诊看板
- 医生工作台
- 报告中心
- 随访提醒与医患消息

患者端包括：

- 报告总览
- 当前报告详情
- 随访日期提示
- 医生消息查看
- 智能问答助手

### 6.2 后端模块

- `auth`：登录验证、JWT 签发
- `upload`：创建 CT 检查记录
- `ai`：创建推理任务、轮询任务状态
- `doctor`：患者分诊、报告中心、报告签发、消息发送
- `patient`：患者报告查询、消息查看
- `chat`：基于报告上下文的患者问答

### 6.3 AI 引擎模块

- `DetectorAdapter`：统一检测器调用入口
- `MockDetector`：用于联调、兜底和演示
- `MonaiBundleDetector`：调用真实肺结节检测模型
- 风险分层逻辑：根据结节尺寸和检测分数生成系统级辅助风险评分
- 后处理逻辑：统一输出结节坐标、直径、体积、风险等级和摘要文本

### 6.4 预览模块

系统支持读取 NIfTI 与 MHD 等格式 CT 文件，生成单张预览图，并在有推理结果时叠加结节标记点。该模块用于课堂展示“AI 检到了哪里”。

## 7. 数据设计

当前课程原型的核心实体包括：

- 用户 `users`
- 检查单 `studies`
- AI 任务 `jobs`
- 分诊表 `patient_triage`
- 报告表 `reports`
- 医患消息表 `doctor_patient_messages`

这些实体足以支撑“医生工作流 + 患者查看 + 随访问答”的完整原型闭环。

## 8. API 设计

系统采用 REST 风格接口，统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

主要接口包括：

- `POST /api/v1/auth/login`
- `POST /api/v1/upload_ct`
- `POST /api/v1/ai/predict/{study_id}`
- `GET /api/v1/ai/jobs/{job_id}`
- `GET /api/v1/doctor/patients`
- `POST /api/v1/doctor/studies/{study_id}/publish_report`
- `GET /api/v1/patient/reports`
- `GET /api/v1/patient/report/{report_id}`
- `POST /api/v1/chat/assistant`

AI Engine 侧提供 `POST /v1/predict` 和 `GET /health` 等接口，实现业务服务与模型服务解耦。

## 9. 可行性分析

### 9.1 技术可行性

前端采用 React，后端采用 FastAPI，AI 检测采用 MONAI 方案，患者问答支持外部大模型兼容接口。这些技术栈成熟且易于教学场景快速实现，技术可行性较高。

### 9.2 工程可行性

系统支持 mock 与真实模型双模式。即使没有 GPU，也可通过 CPU 或 mock 模式跑通完整流程，适合课程答辩环境。

### 9.3 业务可行性

肺结节检测、风险排序、报告解释和随访提醒具有清晰的临床动机，系统闭环较完整，适合展示“AI 医疗产品设计”的课程目标。

## 10. 性能与可靠性分析

1. AI 服务独立部署，减少长耗时推理对业务接口的阻塞。
2. 推理任务采用异步后台执行，前端轮询任务状态。
3. 检测器支持 mock 回退，提高演示稳定性。
4. 系统支持 CPU/GPU 自适应，便于在不同实验环境运行。
5. 影像展示采用轻量化预览图，而非复杂三维渲染，降低实现复杂度。

## 11. 安全与合规

1. 采用 JWT 登录与密码哈希存储。
2. 角色区分为医生、患者、管理员。
3. 患者列表采用脱敏标识显示。
4. 患者问答仅提供解释与建议，不给出确诊性结论。
5. 系统明确标注为课程原型，非临床系统。

## 12. 主要风险与缓解

1. 真实模型在不同环境下依赖复杂。
缓解：保留 mock 检测器和自动回退逻辑，保证系统可演示。

2. 医学影像前端渲染复杂度较高。
缓解：采用单张预览图和标记点展示，优先保证完整流程。

3. 患者问答可能出现不严谨表述。
缓解：限制系统提示词和回答范围，强调以医生意见为准。
